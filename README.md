# school-cream
青衿智眼

## 模型说明（assets 里的模型有什么用）
- 前端会调用 `/api/models`（`assets` 路由）拿到可选模型列表，用于页面下拉选择。
- 真正执行检测的是 `src/main.py` 里的 YOLO 模型加载逻辑。
- `auto` 会优先尝试本地模型（如 `models/best.pt`），没有就回退到 `yolo11n.pt`（由 Ultralytics 自动下载）。

## 为什么 GitHub 传不上去
- `*.pt/*.pth` 模型文件通常很大，超过 GitHub 普通仓库大小限制。
- `__pycache__`、`data/auth.db` 这类本地运行产物也不该进仓库。

## 推荐共享方式
- 代码仓库只放源码，不放大模型本体。
- 首次运行让程序自动下载内置模型（`yolo11n.pt`）。
- 若必须用自定义模型，把模型放网盘/Release，运行后手动放到 `models/`。

## 快速启动
```bash
python src/start.py
```

## 账号保存与权限方案
- 账号必须使用数据库保存，不建议使用本地文件或内存。
- 开发阶段可用 SQLite，准备上线建议迁移到 PostgreSQL 或 MySQL。
- 密码只存哈希值（bcrypt/argon2），禁止明文存储。
- 建议采用 RBAC（基于角色的访问控制），统一在后端做接口鉴权。

### 推荐核心表（最小可用）
- `users`: `id, username, password_hash, role, school_id, class_id, status, created_at`
- `roles`: `admin, teacher, student`
- `classes`: 班级信息与教师关系
- `courses`: 课程、教室、摄像头绑定
- `sessions`: 每次课堂检测会话
- `alerts`: 告警记录（类型、时间、详情、会话ID）
- `metrics`: 课堂统计快照（专注度、行为计数等）

## 三端建设建议（学校管理员 / 教师 / 学生）
- 学校管理员端：用户管理、班级课程管理、全校看板、告警规则配置、模型策略配置。
- 教师端：课堂实时看板、历史记录、告警复盘、报告导出、课堂总结建议。
- 学生端：个人学习趋势、课堂反馈摘要、阶段改进建议（只看本人权限范围）。

## 大模型与智能体功能建议
- 课堂总结智能体：根据一节课的行为统计与告警时间线生成教学复盘建议。
- 异常解释智能体：对异常峰值时段给出可能原因与排查建议（设备/环境/教学节奏）。
- 学生成长助手：根据个人历史趋势生成可执行的学习改进计划。
- 管理问答助手：支持自然语言查询统计数据（如“近两周哪个班手机告警最高”）。

### 落地原则
- 先做“可控智能体”：结构化数据输入 + 模板化输出 + 权限过滤。
- 智能体调用必须经过鉴权与脱敏，避免跨角色数据泄露。
- 优先做一个高价值场景（建议先做“课堂总结智能体”）再逐步扩展。

## 后续开发里程碑（参考）
- 第1阶段（稳定性）：完善账号体系、权限校验、配置管理、启动文档。
- 第2阶段（业务闭环）：打通三端核心页面与接口，完成历史统计与告警入库。
- 第3阶段（智能增强）：接入课堂总结智能体，补充问答与个性化建议能力。
